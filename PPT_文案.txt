tFPS 多人混战 FPS - 技术实现说明

适用场景：PPT 演讲 / 技术分享 / 项目答辩
每段可独立拆为一页幻灯片，小标题即页标题。


一、项目概述

基于 Unreal Engine 5.7 的多人第一人称射击游戏
游戏模式：Deathmatch 混战（FFA），上限 10 人，5 分钟一局
网络架构：Listen Server（主机即玩家），可无缝迁移到 Dedicated Server
纯 C++ 逻辑层 + 蓝图表现层，输入系统使用 Enhanced Input


二、网络架构设计

核心原则：服务端权威（Server-Authoritative）

所有伤害判定、血量变更、道具使用在服务端执行
客户端通过 RPC 发起请求，服务端校验后执行
关键状态（血量/弹药/死亡/背包）通过属性复制同步到所有客户端
背包使用 COND_OwnerOnly 仅复制给本人，防止作弊窥屏
非关键表现（枪口火焰/音效/弹壳）由本地事件驱动，零延迟

三种同步机制的分工：
  属性复制 -> 持续状态（血量/弹药/分数）
  RPC -> 一次性动作（开火/换弹/拾取/使用道具）
  事件广播 -> 表现层通知（开火特效/命中反馈/UI 刷新）


三、核心类架构

AFPSGameMode          [仅服务端]   计时、胜负判定、玩家生成
AFPSGameState         [复制全端]   比赛时间、阶段
AFPSPlayerState       [复制全端]   击杀/死亡/伤害/带出价值
AFPSPlayerController  [Svr+Owner]  重生 RPC、计分板、HUD 管理
AFPSCharacter         [复制]       血量/移动/死亡复活/武器管理/背包
AFPSWeapon            [复制Actor]  射击/换弹/弹药（服务端权威）
AFPSPickup            [复制Actor]  地上道具拾取物（复制销毁）
AFPSWeaponPickup      [复制Actor]  地上武器拾取物
AFPSSubmissionPoint   [复制Actor]  随机开放关闭的物品提交点
AFPSPickupSpawner     [仅服务端]   周期性生成掉落物
UFPSInventoryComponent[组件]       背包逻辑（COND_OwnerOnly）
UFPSInteractionComponent[组件]     交互目标管理
UFPSItemDef           [DataAsset]  道具数据多态基类

设计原则：
  逻辑全在 C++，蓝图只做表现（UI/动画/特效/音效）
  组件化拆分：背包、交互、HoT 各自独立 Component
  接口驱动：道具子类 override ServerUseItem，背包不关心道具类型


四、输入系统

技术选型：Enhanced Input + Axis2D 架构

所有输入资产（IA/IMC）由蓝图配置，C++ 只绑定处理函数
移动和视角使用 Axis2D 值类型（WASD -> FVector2D，Mouse -> FVector2D）
Boolean 型输入（Shift/右键/LMB/R/F/E/Tab/1/2）走 Started/Completed 事件
鼠标滚轮用 Axis1D，正值滚下一项、负值滚上一项

关键坑点与解决方案：
  问题：BeginPlay 时 Controller 可能尚未 Possess，IMC 注册失败
    -> SetTimerForNextTick 延迟一帧再注册

  问题：动态创建的 UInputModifier 在 Listen Server 上跨帧被 GC
    -> 改用蓝图序列化资产，不 C++ NewObject

  问题：RequestRebuildControlMappings 会丢弃动态 Modifier
    -> 不调用此函数

计分板采用独立 IMC + 更高优先级（1 > 0），两段式鼠标锁定：
  Tab 按下 -> 显示面板，纯键盘操作（不影响战斗）
  面板开时点鼠标 -> 锁定鼠标进 UI（点排序），禁开火/瞄准


五、武器与射击系统

服务端权威碰撞检测：

  客户端按 LMB -> ServerStartFire RPC
    -> 服务端 StartFire -> FireTimer 每 FireRate 调 HandleFire
      -> 从 Pawn 视角 LineTrace（专属 Weapon 碰撞通道）
        - 命中角色 -> ApplyDamage（服务端权威）
        - 命中环境 -> MulticastHitConfirmed（各端分发特效）

命中反馈三通道设计：

  MulticastHitConfirmed（Unreliable，武器上发）
    -> 各端 DispatchHitFX：
      - 本机控制攻击者     -> OnDealtHit（命中标记 UI / 命中音）
      - 本机控制受害者     -> OnReceivedHit（受伤镜头模糊 / 抖动）
      - 无条件             -> OnHitWorld（冲击特效 / 弹孔）

专属碰撞通道（Weapon Channel）：
  与 Visibility 隔离，避免 AI 视线/鼠标拾取干扰
  角色 Mesh 设 Block（Physics Asset 简化体），胶囊体设 Ignore
  bTraceComplex=false，走 Physics Asset 而非逐三角面，性能优

开火表现与逻辑分离：
  C++ 只管权威 LineTrace + 伤害
  蓝图监听 OnFireStarted/Stopped 事件驱动本地 timer + 特效
  本地 timer 零延迟，不受网络波动影响；服务端 timer 独立
  本地可能多播 1-2 发特效，对玩家感知无影响


六、弹药系统

设计：弹夹归武器 + 备弹归背包 + 口径匹配

CurrentAmmo（弹夹）在 AFPSWeapon 上 Replicated
备弹作为 UFPSAmmoItemDef 条目存在玩家背包中
武器 AcceptedCaliber（FName）匹配弹药 Caliber
换弹时从背包末尾向前逐格取弹，耗尽条目自动删格

分级体系：
  弹药按 (Caliber, Tier) 区分：5.56 T1~T5, 7.62 T1~T5, 9mm T1~T5
  不同 (口径, 等级) 的组合 = 不同的 DataAsset -> 天然不堆叠
  LoadedAmmoDef 记录当前弹夹装填的弹药类型，决定单发伤害加成
  伤害 = 武器基础 DamageAmount + 弹药 BaseDamage

换弹流程：
  按 R -> 本地 CanReload 预判 -> ServerReload RPC
    -> StopFire + bIsReloading=true -> ReloadTimer(ReloadTime)
      -> ConsumeAmmo(AcceptedCaliber, Needed) -> 补弹入夹
      -> 更新 LoadedAmmoDef -> MarkItemsDirty -> UI 刷新


七、道具/背包系统

三层身份分离（核心架构决策）：

  Pickup（地上 Actor）: 地图可见、可捡、捡走 Destroy
  ItemDef（DataAsset）: 道具数据：名字/图标/数值/使用效果
  InventoryEntry（UStruct）: 背包一条记录：ItemDef 指针 + 数量/耐久

混合存储模型：
  可堆叠物品（bUsesDurability=false）：用 Count 字段，同类堆叠
  带耐久物品（bUsesDurability=true）：用 Durability 字段，每件独立

多态道具效果（方案 A）：
  基类 UFPSItemDef 定义 virtual 接口：
    GetItemUseType() / GetCurrentValue() / ServerCanUse() / ServerUseItem()

  五种子类各 override ServerUseItem：
    HealItemDef          : 点即回血（即时治疗）
    ChanneledHealItemDef : 前摇后逐跳回血至承诺量（可取消）
    HoTItemDef           : 使用耗时后激活持续回血 buff
    ValuableItemDef      : 不可使用，仅提交计分
    AmmoItemDef          : 换弹消耗，不可直接使用

  背包组件只调 ItemDef->ServerUseItem，完全不关心道具类型。
  新增道具类型只需新建 DataAsset 子类，不改背包代码。

异步道具使用流程（ChanneledHeal / HoT）：
  客户端 UseInventoryItem -> ServerUseInventoryItem RPC
    -> Inventory->BeginItemUse 扣耐久
    -> Character->StartItemUseProcess：
      ChanneledHeal: UseTime 前摇后逐跳回血 -> Complete
      HoTApply: UseTime 后激活 HoT buff（独立 timer）
    -> F 键取消（已扣耐久不退还）
    -> 使用期间 IsActionLocked 禁开火/瞄准/奔跑/换弹/拾取

空间占格升级预留：
  当前为"计数网格"，每个道具 1 格。已留好 GridWidth/GridHeight 接口
  唯一替换点 = FindSlotFor()，改为装箱算法即可支持不同尺寸道具
  UI 从 UniformGridPanel 改为 CanvasPanel 按坐标定位


八、交互管理系统

设计目标：F 键面对多种交互目标时，玩家可用滚轮切换选中

交互类型与优先级：
  WeaponPickup       优先级 100 : "Swap weapon"（武器拾取优先）
  SubmissionPoint    优先级 50  : "Submit valuables"
  Pickup             优先级 0   : "Pick up {ItemName}"

生命周期（各端本地独立维护）：
  进入范围 -> RegisterInteraction(Source, Type, Prompt, Priority)
  离开范围 -> UnregisterInteraction(Source)
  首个条目自动选中；删除条目时自动调整选中索引

不复制：每个端根据本地 overlap 事件独立维护交互列表


九、武器拾取与掉落

武器不进背包，独立于道具拾取系统：

拾取流程（按 F，武器优先于道具）：
  0 枪 -> 装备为主武器
  1 枪 -> 装备为副武器（不切换活跃槽）
  2 枪 -> 与当前活跃槽武器交换，旧武器掉落到地上

死亡掉落：
  比较主副武器的 WeaponValue，掉落价值更高的那把
  若掉的恰好是活跃武器且另一把存活 -> 自动切槽
  若唯一武器掉落 -> 复活时重新生成默认武器

AFPSWeaponPickup 记录掉落时的弹夹剩余弹药和类型，拾取后保留


十、物品提交系统

场景手摆的提交点 Actor，随机开放/关闭：

状态机（仅服务端）：
  Closed -> 随机 30-120s -> Open -> 随机 15-45s -> Closed -> 循环

提交流程：
  一键提交：走近开放提交点按 F -> 遍历背包贵重品 -> 累加价值 -> 删格
  单件提交：背包右键"提交"按钮（仅开放提交点范围内显示）
  价值累加到 PlayerState.CarryValue，计入战绩面板

关键设计：
  状态通过 OnRep 复制到所有端 -> 蓝图切换灯光/粒子/音效
  服务端 GetTimerHandle 排程状态切换，不轮询


十一、物品生成系统

设计：事件驱动而非轮询

生成循环：
  生成 Pickup -> 等待被拾取 -> OnDestroyed 委托触发
    -> 进入冷却倒计时（BaseSpawnInterval +/- 随机偏移）
    -> 下一轮生成

Tier 递进机制：
  BaseSpawnTier 线性增长到 MaxSpawnTier（TierRampTime 后）
  使游戏后期生成的道具等级逐渐提高

防堆叠：
  CurrentSpawnedPickup 有效性检查（有则跳过）
  SpawnCheckRadius 重叠测试（兜底）
  SpawnActorDeferred + 设 ItemDef + FinishSpawning
    确保蓝图 Construction Script 能读到 ItemDef

权重抽选：
  SpawnTable 条目 x BaseWeight x BiasDegree（若匹配 BiasItemClass）
  跳过 Tier 高于当前 MaxTier 的条目


十二、主副武器系统

角色最多持两把枪：槽 0（主）+ 槽 1（副），均 Replicated
ActiveWeaponSlot 决定当前开火/换弹操作哪一把
切换时自动停止旧武器的开火/瞄准状态
蓝图可见性由 C++ SetActorHiddenInGame 控制，无需蓝图额外处理
OnWeaponEquipped / OnSecondaryWeaponEquipped 分别挂载到 Arm 组件


十三、战绩面板系统

C++ 逻辑全包 + 蓝图摆控件：

数据来源：AFPSPlayerState（Kills/Deaths/TotalDamage/CarryValue）
C++ 基类 UFPSScoreboardWidget 负责收集、排序、刷新
可切换排序键：击杀数 / 总伤害 / 带出价值
蓝图继承后只做 UI 铺行，数据由 GetSortedRows() 提供

两段式鼠标锁定：
  Tab 按下 -> 显示面板（GameOnly 模式，不影响战斗）
  面板开时点鼠标 -> 切 GameAndUI + 弹鼠标（点排序按钮）
  Tab 松开 -> 隐藏 + 复位

与背包互斥：计分板打开期间不能开背包，反之亦然


十四、死亡与复活

血量 <= 0 -> 服务端处理：
  攻击者 PlayerState.Kills++
  受害者 PlayerState.Deaths++
  受害者 Die() -> 禁用移动/碰撞，掉落武器
  3 秒后 -> ChoosePlayerStart 随机选点 -> RestartPlayer 重生

复活时：
  满血 + 重新生成默认武器（若无武器存活）
  满弹夹 + LoadedAmmoDef 复位（伤害加成归零）
  清除 HoT buff 和使用道具状态


十五、技术亮点总结

1. 网络架构
   - 服务端权威 + 属性复制 + RPC 三层配合
   - 表现与逻辑严格分离，本地事件零延迟，权威逻辑不妥协

2. 道具系统多态设计
   - ItemDef DataAsset 子类 override ServerUseItem
   - 新增道具类型只需新建子类，不改背包/拾取/UI 代码
   - 混合存储（堆叠/耐久）用同一套 FInventoryEntry

3. 事件驱动架构
   - 开火表现：OnFireStarted/Stopped -> 蓝图本地 timer -> 特效（零延迟）
   - 命中反馈：一条 Multicast -> 三路分发（OnDealtHit/OnReceivedHit/OnHitWorld）
   - 生成器：OnDestroyed 委托链，不轮询

4. 组件化拆分
   - 背包、交互、HoT 各自独立 Component
   - 战绩面板挂 Controller 不挂 Pawn（死亡换 Pawn 不丢）

5. 输入系统健壮性
   - 解决 Listen Server 下动态 Modifier GC、IMC 注册时序等 UE 特有坑
   - 计分板独立 IMC + 优先级隔离 + 两段式鼠标锁定

6. 扩展预留
   - 背包空间占格接口已就绪（GridWidth/GridHeight），升级装箱算法不改逻辑
   - Listen Server 架构无缝迁移 Dedicated Server
   - 套娃背包、爆头判定、部位伤害等下一步方向已规划
